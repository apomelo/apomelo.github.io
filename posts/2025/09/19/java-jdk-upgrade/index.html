Java JDK Upgrade

发表于 2025-09-19 分类于 java 阅读次数：

简介

本文梳理了 Java 8 到 Java 25 各版本的核心特性演进，帮助开发者了解 Java 的发展脉络，为 JDK 升级决策提供参考。

为什么要升级 JDK？

许多项目仍在使用 Java 8，但新版本 JDK 带来了显著的改进：

维度 升级收益
性能 新版 GC（ZGC/Shenandoah）实现亚毫秒级暂停，JIT 编译优化提升 10-20% 吞吐量
并发 虚拟线程（Java 21）支持百万级轻量线程，彻底改变高并发编程模型
语法 var、Record、模式匹配、文本块等特性大幅减少样板代码
安全 持续的安全补丁和漏洞修复，旧版本已停止公开更新
生态 主流框架（Spring 6+、Quarkus、Micronaut）要求 Java 17+

推荐升级路径

  • Java 8 → Java 17：稳定 LTS，企业首选，Spring Boot 3.x 最低要求
  • Java 17 → Java 21：获得虚拟线程，高并发场景强烈推荐
  • Java 21 → Java 25：最新 LTS，作用域值 + 内存优化

Java 8

Java 8 发布于 2014 年 3 月 18 日，是 Java 历史上最重要的版本之一，也是使用最广泛的 LTS 版本。Java 8 引入了函数式编程特性，彻底改变了 Java 的编程范式。

  • Lambda 表达式

  • 函数式接口
    • 新增 @FunctionalInterface 注解，内置 Function、Predicate、Consumer、Supplier 等函数式接口：
  • Stream API
    • 引入流式处理，支持链式操作和并行处理：
  • 方法引用
    • 使用 :: 操作符引用方法：
  • Optional 类
    • 优雅处理空值，避免 NullPointerException：
  • 接口默认方法和静态方法
    • 接口可以包含带实现的方法：
  • 新的日期时间 API（java.time）
    • 引入不可变的日期时间类，替代 Date 和 Calendar：
  • CompletableFuture
    • 增强的异步编程支持：
  • 重复注解和类型注解
    • 允许在同一位置多次使用相同注解。
  • Nashorn JavaScript 引擎
    • 内置 JavaScript 引擎（已在 Java 15 移除）。
  • PermGen 移除
    • 永久代被元空间（Metaspace）取代，使用本地内存。

Java 9

Java 9 发布于 2017 年 9 月 21 日。作为 Java 8 之后 3 年半才发布的新版本，Java 9 带来了很多重大的变化，其中最重要的改动是 Java 平台模块系统的引入，其他还有诸如集合、Stream 流等等

  • 模块化系统（Project Jigsaw）

    • 将 JDK 拆分为多个模块，允许开发者根据需要选择性地引入模块，减少应用程序的体积和复杂性。

  • JShell：交互式编程环境

    • 允许开发者在命令行中编写和执行 Java 代码，方便学习和测试。

  • G1垃圾回收器成为默认垃圾回收器

    • 提供更好的性能和更低的延迟，适合大内存应用。

  • 快速创建不可变集合

    • 通过 List.of(), Set.of(), Map.of() 等方法快速创建不可变集合。

  • String 存储结构优化

    • Java 8 及之前的版本，String 一直是用 char[] 存储。在 Java 9 之后，String 的实现改用 byte[] 数组存储字符串，节省了空间。

  • 接口私有方法

    • 在 Java 9 之前，接口只能包含抽象方法和默认方法。Java 9 引入了接口私有方法，允许在接口中定义私有方法，以便在默认方法中复用代码。

  • try-with-resources 增强

    • Java 9 引入了对 try-with-resources 语句的增强，允许在 try 块中使用已经声明的资源，而不需要在 try 块中重新声明资源。
      • 在 Java 9 之前，我们只能在 try-with-resources 块中声明变量：
      • 在 Java 9 之后，在 try-with-resources 语句中可以使用 effectively-final 变量。

  • 流API增强

    • Java 9 对流API进行了增强，新增了 takeWhile(), dropWhile(), ofNullable(), 重载的iterate() 等方法，提供了更多的流操作功能。
    • takeWhile(Predicate<? super T> predicate)：从流头开始保留满足条件的元素，一旦遇到第一个不满足的元素(不包含这个元素)就停止，返回已保留的前缀。
    • dropWhile(Predicate<? super T> predicate)：从流头开始丢弃满足条件的元素，一旦遇到第一个不满足的元素(包含这个元素)就停止丢弃，返回从该位置到流尾的剩余元素。
    • ofNullable(T t)：返回一个流，如果参数 t 不为 null，则返回包含该元素的流；否则返回一个空流。
    • iterate​(T seed, Predicate<? super T> hasNext, UnaryOperator next)：返回一个流，从种子元素开始，按照给定的 next 函数生成后续元素，直到 hasNext 返回 false 为止。

  • Optional 类增强

    • Java 9 对 Optional 类进行了增强，新增了 ifPresentOrElse(), or(), stream() 等方法，提供了更多的操作选项的功能。

  • 进程 API

    • Java 9 引入了新的进程 API java.lang.ProcessHandle，允许开发者更方便地管理和控制操作系统进程。

  • 改进的JVM性能和监控工具

Java 10

Java 10 发布于 2018 年 3 月 20 日。这是 Java 采用新的发布周期（每 6 个月发布一个新版本）后的第一个版本。Java 10 带来了一些重要的新特性，其中最引人注目的是局部变量类型推断。

  • 局部变量类型推断（var）
    • Java 10 引入了 var 关键字，允许编译器根据初始化表达式自动推断局部变量的类型，减少冗余代码。
    • 注意：var 只能用于局部变量，不能用于方法参数、返回类型、字段或 catch 块中的异常类型。
  • 不可变集合增强
    • 新增 List.copyOf(), Set.copyOf(), Map.copyOf() 方法，用于创建集合的不可变副本。
    • Collectors 新增 toUnmodifiableList(), toUnmodifiableSet(), toUnmodifiableMap() 方法。
  • Optional 增强
    • 新增 orElseThrow() 无参方法，当 Optional 为空时抛出 NoSuchElementException。
  • G1 垃圾回收器并行 Full GC
    • G1 垃圾回收器的 Full GC 改为并行执行，提高了 Full GC 的性能。
  • 应用程序类数据共享（Application Class-Data Sharing）
    • 扩展了现有的类数据共享（CDS）功能，允许应用程序类被放置在共享存档中，以缩短启动时间和减少内存占用。
  • 线程本地握手（Thread-Local Handshakes）
    • 允许在不执行全局 VM 安全点的情况下对单个线程执行回调，提高了 JVM 的性能和可诊断性。
  • 基于时间的版本发布
    • Java 10 开始采用基于时间的版本发布模式，每 6 个月发布一个新版本，版本号格式为 $FEATURE.$INTERIM.$UPDATE.$PATCH。
  • 根证书
    • 在 JDK 中提供一组默认的根证书颁发机构（CA）证书，使 OpenJDK 构建对开发人员更具吸引力。
  • 实验性的基于 Java 的 JIT 编译器（Graal）
    • 将基于 Java 的 JIT 编译器 Graal 作为 Linux/x64 平台上的实验性 JIT 编译器。

Java 11

Java 11 发布于 2018 年 9 月 25 日。这是继 Java 8 之后的第一个长期支持版本（LTS），也是很多企业从 Java 8 升级的首选目标版本。Java 11 带来了许多实用的新特性和 API 增强。

    • 新增了一系列实用的字符串处理方法：
  • 集合转数组增强

  • 文件读写 API 增强
    • 新增 Files.readString() 和 Files.writeString() 方法，简化文件读写操作：
  • Optional 增强
    • 新增 isEmpty() 方法，与 isPresent() 相反：
  • Lambda 参数的局部变量语法
    • 允许在 Lambda 表达式的参数中使用 var，主要用于添加注解：
  • HTTP Client API（正式版）
    • Java 9 引入的 HTTP Client API 在 Java 11 中正式发布，支持 HTTP/1.1 和 HTTP/2，支持同步和异步请求：
  • 直接运行 Java 文件
    • 可以直接使用 java 命令运行单个 Java 源文件，无需先编译：
  • ZGC 垃圾回收器（实验性）
    • 引入了 ZGC（Z Garbage Collector），一个可扩展的低延迟垃圾回收器，目标是将 GC 暂停时间控制在 10ms 以内。
  • Epsilon 垃圾回收器（实验性）
    • 一个”无操作”的垃圾回收器，不执行任何垃圾回收操作，适用于性能测试和短生命周期应用。
  • Flight Recorder（飞行记录器）
    • Java Flight Recorder（JFR）开源，提供低开销的数据收集框架，用于诊断 Java 应用程序和 JVM。
  • 移除和废弃的功能
    • 移除了 Java EE 和 CORBA 模块（如 java.xml.ws, java.xml.bind 等）。
    • 废弃了 Nashorn JavaScript 引擎。
    • 移除了 Thread.destroy() 和 Thread.stop(Throwable) 方法。

Java 14

  • NullPointerException 改进
    • 改进了 NullPointerException 的错误信息，精确指出哪个变量为 null：
  • JFR 事件流

Java 15

Java 15 发布于 2020 年 9 月 15 日，是短期支持版本，继续完善多项预览特性。

  • 文本块（正式版）
    • 文本块在经历两次预览后，在 Java 15 中正式成为标准特性。
  • Sealed Classes（密封类，预览）
    • 引入密封类，限制哪些类可以继承或实现：
  • instanceof 模式匹配（二次预览）
    • 继续作为预览特性，无重大变化。
  • Record 类（二次预览）
    • 继续作为预览特性，支持本地 Record 类定义。
  • 隐藏类（Hidden Classes）
    • 引入隐藏类，供框架在运行时生成类使用，这些类不能被其他类直接使用，且有较短的生命周期。
  • ZGC 和 Shenandoah 正式发布
    • ZGC 和 Shenandoah 垃圾回收器从实验性特性转为正式特性。
  • EdDSA 数字签名算法
    • 实现了 Edwards-Curve 数字签名算法（EdDSA），提供更好的性能和安全性。
  • 移除 Nashorn JavaScript 引擎
    • 正式移除了在 Java 11 中废弃的 Nashorn JavaScript 引擎。
  • 废弃偏向锁
    • 废弃了偏向锁（Biased Locking），因为维护成本高且收益有限。

  • 日期时间格式化增强

重要特性介绍

虚拟线程

Java 虚拟线程最早源自 Project Loom，Oracle 在 2017 年正式提出。JDK19 引入预览功能，最后在 JDK21 正式启用。

为什么 Java 迟迟没有支持虚拟线程

历史原因

  • Java 一开始的线程模型就是系统线程映射模型：Thread 对应一个原生操作系统线程。
  • 硬件和 JVM 技术条件有限的情况下，做轻量级线程会让垃圾回收、同步、调度、异常处理等问题复杂化。
  • Java 生态太庞大，几乎所有框架、库、监控工具等都建立在现有线程模型之上，属实有点改不动。

Java 已有线程模型的优缺点

Java 的线程池设计 ThreadPoolExecutor 非常成熟，能满足绝大多数场景。同时配合 ForkJoinPool 在 CPU 密集任务中表现很好。

当然它的缺点也是非常明显的：

  • 线程栈空间固定，默认 1M 左右（可以通过启动参数 -Xss 指定）创建大量线程需要消耗大量的内存。
  • 上下文切换开销比较高，如果线程数在几千这个数量级的时候，调度成本会比较高。

当前的互联网环境，我们希望每个 JVM 实例，能承载高并发请求，现在传统的线程模型就会显得过于重。

Java 虚拟线程池设计

Java 虚拟线程的设计目标是：用极低的内存开销创建大量线程，且能很好地与现有 Java 生态兼容。

所以虚拟线程目前是可选项，用户需要在创建线程和线程池的时候，主动选择创建传统线程或者虚拟线程。

线程模型

虚拟线程与平台线程关系（M:N 模型）

  • 虚拟线程（Virtual Thread）：用户态，创建/销毁极轻量，栈按需增长（初始几 KB），阻塞时可卸载（unmount）释放载体线程。
  • 平台线程（Platform/Carrier Thread）：映射到操作系统内核线程，由 OS 调度，数量通常≈CPU 核心数，运行虚拟线程的实际承载者。
  • 在 Windows 和 Linux 等主流操作系统中，Java 线程采用的是一对一的线程模型，也就是一个平台线程对应一个系统内核线程。
  • 调度机制：大量虚拟线程排队；被分配到空闲平台线程上运行；遇到可挂起的阻塞（I/O、park、sleep、同步等待）时，栈封装为 continuation，虚拟线程从平台线程上“脱离”，平台线程再去运行其他虚拟线程；不可卸载的阻塞（pinning，例如持有某些监视器执行阻塞本地调用）会占住平台线程。

核心特性概览

  1. M 个虚拟线程 -> N 个平台线程（通常 N << M）
  2. 可挂起阻塞 ≈ 不占用平台线程
  3. Pinning 场景需要避免：
  • 在 synchronized 保护的临界区内执行阻塞 I/O / sleep / park
  • 调用不支持虚拟线程卸载的本地/阻塞库方法
  1. 目标：以“线程即结构”简化并发，而不是回调/显式任务拆分。

简化调度循环

  1. 创建虚拟线程 -> 放入就绪队列
  2. 平台线程拉取 -> 挂载并执行
    3a) 正常完成 -> 释放
    3b) 可挂起阻塞 -> 卸载，返回队列等待唤醒
    3c) Pinning 阻塞 -> 占用平台线程直至恢复（需谨慎避免）

实践建议

  • 优先使用结构化并发（StructuredTaskScope）管理虚拟线程生命周期
  • 避免在 synchronized 大块中执行阻塞操作
  • 评估第三方库是否存在本地阻塞调用（Netty、JDBC 驱动等）
  • 监控 pinning（可用 JFR 事件 VirtualThreadPinned）

小结：虚拟线程通过“可卸载 + Continuation”使阻塞不再等于占用载体线程，从而以直观的同步代码风格实现大规模并发。

线程栈

Java 虚拟线程的栈空间是动态分配的，初始时只有几 KB 大小，当虚拟线程执行过程中需要更多栈空间时，JVM 会自动扩展栈空间。当虚拟线程执行完成后，JVM 会回收栈空间，从而节省内存资源。
