简介

Prometheus 是一个开源系统监控和报警工具包，最早由 SoundCloud 开发，现在是 CNCF（Cloud Native Computing Foundation）的一个项目。它的设计目标是可靠性和高效性，特别适用于动态云环境下的系统和服务监控。

核心特性

  1. 多维数据模型： Prometheus 使用多维数据模型，通过时间序列来存储数据。每个时间序列由一个度量名称和一组键值对的标签来标识。
  2. 强大的查询语言（PromQL）： Prometheus 提供了一种灵活而强大的查询语言，PromQL，用于实时查询和分析存储的数据。
  3. 无需依赖分布式存储： Prometheus 的设计目标是通过单个服务器节点提供可靠的服务，这使其配置和管理更为简单。尽管它可以通过联邦集成多个 Prometheus 服务器，但它本身不依赖于分布式存储系统。
  4. 高效的数据存储： 数据以时间序列的形式存储，并通过内存和磁盘的组合方式进行高效存储和检索。
  5. 可视化支持： Prometheus 提供内置的图形和仪表盘功能，同时与 Grafana 等外部可视化工具高度集成。
  6. 灵活的告警管理： Prometheus 自带 Alertmanager，可以管理告警规则，处理告警并将其路由到不同的通知渠道（如电子邮件、Slack、PagerDuty 等）。

工作原理

Prometheus 通过 HTTP 协议定期抓取被监控目标的指标数据。这些目标可以是物理主机、虚拟机、数据库、应用程序等。被监控目标需要暴露一个 /metrics 端点，Prometheus 会通过这个端点获取指标数据。

基本组件

  1. Prometheus 服务器： 负责抓取和存储时间序列数据，处理查询请求。
  2. Exporters（导出器）： 用于暴露指标数据的工具。不同类型的系统和应用程序可能需要不同的导出器。例如，Node Exporter 用于暴露 Linux 系统的硬件和操作系统指标。
  3. Pushgateway： 用于短期任务的指标推送。通常情况下，Prometheus 是通过 pull 模型抓取数据，但对于某些短期任务，推送模型更为合适。
  4. Alertmanager： 处理来自 Prometheus 服务器的告警，支持告警的分组、抑制和路由，并将告警发送到各种通知渠道。
  5. 客户端库： 提供给应用程序开发者用于在代码中直接暴露自定义指标。Prometheus 提供多种语言的客户端库，包括 Go、Java、Python 等。

典型使用场景

  1. 系统监控： 监控主机和容器的性能指标，如 CPU 使用率、内存使用率、磁盘 I/O 等。
  2. 应用程序监控： 监控应用程序的性能和健康状况，例如 HTTP 请求的响应时间、错误率、数据库查询时间等。
  3. 业务指标监控： 监控业务相关的指标，例如网站的活跃用户数、订单处理量、支付成功率等。

配置简介

Prometheus 配置文件（通常为 prometheus.yml）定义了如何配置 Prometheus 服务器，包括抓取（scrape）配置、告警规则（alerting rules）、全局设置和存储配置等。下面是一个典型的 prometheus.yml 配置文件示例，以及对每个部分的详细解析：

全局配置 (global)

global 部分包含 Prometheus 的全局设置，适用于所有的 scrape 和告警配置。

  • scrape_interval: 定义了 Prometheus 默认的抓取目标的时间间隔（默认每 15 秒）。
  • evaluation_interval: 定义了 Prometheus 评估规则（包括告警规则和记录规则）的时间间隔（默认每 15 秒）。
  • scrape_timeout: 定义了抓取的超时时间，如果一个抓取任务超过这个时间，将被终止（默认 10 秒）。

告警配置 (alerting)

alerting 部分定义了告警管理器（Alertmanager）的配置，Prometheus 将触发的告警发送给 Alertmanager。

  • alertmanagers: 指定了 Alertmanager 的地址，Prometheus 会将触发的告警发送到这些 Alertmanager 实例。

规则文件 (rule_files)

rule_files 部分指定了 Prometheus 使用的规则文件，包括告警规则和记录规则。

  • rule_files: 列出了包含告警规则和记录规则的文件。Prometheus 会加载并评估这些文件中的规则。

抓取配置 (scrape_configs)

scrape_configs 部分定义了 Prometheus 如何抓取监控目标（targets）。每个抓取配置定义了一个 job。

  • job_name: 定义了抓取任务的名称，用于标识和分类不同的抓取任务。
  • static_configs: 指定了静态配置的抓取目标列表。

scrape_configs 中的配置选项

  • job_name: 每个抓取任务的唯一名称。
  • static_configs: 静态配置，用于定义一组固定的抓取目标。
    • targets: 定义了抓取目标的地址列表。
  • relabel_configs: 重新标签配置，用于修改抓取的目标标签。

  • relabel_configs: 用于在抓取之前或之后修改标签。可以通过正则表达式匹配和替换标签。

小结

Prometheus 配置文件主要包括以下几个部分：

  • 全局配置：定义默认的抓取和评估间隔时间。
  • 告警配置：指定 Alertmanager 的地址。
  • 规则文件：列出包含告警规则和记录规则的文件。
  • 抓取配置：定义抓取任务，包括抓取目标和标签重写规则。

这些配置可以帮助 Prometheus 正确地抓取监控目标、评估规则并处理告警。

规则文件

在 Prometheus 中，规则文件用于定义记录规则（Recording Rules）和告警规则（Alerting Rules）。规则文件通常使用 YAML 格式进行编写，并在 prometheus.yml 配置文件中通过 rule_files 指令包含进来。下面是一个典型的规则文件示例，以及对其各部分的详细解析：

规则文件结构

规则文件的基本结构包括一个或多个 groups，每个组包含一组规则（规则可以是记录规则或告警规则）。

  • groups: 包含一个或多个组，每个组有一个名称和一组规则。
    • name: 组的名称，用于标识和分类规则。
    • rules: 组中的规则列表。

记录规则（Recording Rules）

记录规则用于预计算复杂的查询表达式，并将其结果存储为新的时间序列，以便在后续查询中使用。

  • record: 定义新的时间序列的名称。
  • expr: PromQL 表达式，用于计算新的时间序列值。
  • labels: 为新的时间序列添加标签（可选）。
  • annotations: 为新的时间序列添加注释（可选）。

在这个示例中，记录规则计算每个 job 的 http_inprogress_requests 的总和，并将结果存储为 job:http_inprogress_requests:sum。

告警规则（Alerting Rules）

告警规则用于定义在特定条件下触发的告警，并发送给告警管理系统（如 Alertmanager）。

  • alert: 告警名称。
  • expr: PromQL 表达式，当该表达式为真时触发告警。
  • for: 持续时间，如果表达式在此时间内持续为真，则触发告警（可选）。
  • labels: 为告警添加标签，用于分类和筛选告警（可选）。
  • annotations: 为告警添加注释，通常用于提供告警的详细信息和上下文（可选）。

在这个示例中，当 job="myservice" 的 request_latency_seconds:mean5m 超过 0.5 且持续时间超过 10 分钟时，触发 HighRequestLatency 告警。

什么是 CRD？

CRD，全称 Custom Resource Definition（自定义资源定义），是 Kubernetes 提供的一种机制，用于扩展 Kubernetes API。通过 CRD，用户可以创建自己的资源类型，类似于 Kubernetes 原生资源（如 Pods、Services 等），从而增强 Kubernetes 的功能和灵活性。

CRD 的基本概念

  1. Custom Resource（CR）： 自定义资源，是根据 CRD 定义的一种具体资源实例。CR 就像 Pods 和 Services 一样，通过 Kubernetes API 进行创建和管理。
  2. Custom Resource Definition（CRD）： 自定义资源定义，是用于定义新的自定义资源类型的 YAML 文件。CRD 描述了自定义资源的名称、属性和行为。

创建 CRD 的步骤

  1. 定义 CRD： 编写 CRD 的 YAML 文件，描述自定义资源的 API 版本、属性和验证规则。
  2. 应用 CRD： 使用 kubectl apply 命令将 CRD 应用到 Kubernetes 集群。
  3. 创建自定义资源： 一旦 CRD 创建成功，就可以根据 CRD 创建自定义资源实例。

定义 CRD

  • apiVersion: CRD 的 API 版本。
  • kind: 表示资源类型为 CustomResourceDefinition。
  • metadata: 元数据，包含 CRD 的名称。
  • spec: 定义 CRD 的详细规格。
    • group: 自定义资源所属的 API 组。
    • versions: 定义自定义资源的多个版本。
      • name: 版本名称。
      • served: 是否提供该版本的服务。
      • storage: 是否将该版本作为存储版本。
      • schema: 自定义资源的 OpenAPI v3 模式，定义属性和类型。
    • scope: 定义自定义资源的作用域（Namespaced 或 Cluster）。
    • names: 定义自定义资源的名称，包括复数名、单数名、资源种类和简写。

创建自定义资源

  • apiVersion: 使用自定义资源的 API 版本。

  • spec: 自定义资源的具体规格，包括数据库名称、用户和密码等属性。

注意事项

  1. 权限管理： 确保 Prometheus 和 Alertmanager 有适当的权限访问所需的资源。通常，通过设置适当的 serviceAccount 并将其绑定到 ClusterRole 和 ClusterRoleBinding。
  2. 标签匹配： ServiceMonitor 的 selector 标签应与要监控的服务的标签匹配，以确保 Prometheus 能够正确地抓取指标。
  3. 资源限制： 为 Prometheus 和 Alertmanager 设置合理的资源请求和限制，以确保它们在集群中稳定运行。
  4. 数据持久化： 配置持久化存储卷（PVC）来存储 Prometheus 的时间序列数据，防止数据丢失。
  5. 高可用性： 配置 Prometheus 和 Alertmanager 的副本数量（replicas）以实现高可用性。
  6. 网络策略： 考虑使用网络策略（NetworkPolicy）来限制 Prometheus 和其他组件之间的流量，以增强安全性。

小结

通过使用 Prometheus Operator 和 CRD，可以轻松地在 Kubernetes 集群中部署和管理 Prometheus、Alertmanager 以及相关的监控规则。关键在于正确配置 CRD，并确保 Prometheus 和其他组件能够相互通信并访问所需的资源。这样，便能实现强大的监控和告警功能，确保系统的稳定性和性能。
