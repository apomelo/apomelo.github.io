网络基础

http 是什么

HTTP 协议是 HyperText Transfer Protocol, 超文本传输协议

WWW: World Wide Web

http 协议原理

  1. HTTP 协议工作于客户端-服务端架构上。浏览器作为 HTTP 客户端通过 URL 向 HTTP 服务端即 WEB 服务器发送所有请求。
  2. HTTP 是无连接的：无连接是指每次连接只处理一个请求。
  3. HTTP 是媒体独立的：只要客户端和服务器知道如何处理的数据内容，任何类型的数据都可以通过 HTTP 发送。
  4. HTTP 是无状态的：无状态是指两个执行成功的请求之间没有关系。
  5. HTTP 是有会话的：HTTP 的头部扩展 Cookies 就可以解决无状态问题，创建一个会话让每次请求都能共享相同的上下文信息，达成相同的状态。

http 一次请求过程

  1. 用户在浏览器中键入需要访问网页的 URL 或者点击某个网页中链接
  2. 在 HTTP 开始工作前，客户端首先会通过 TCP/IP 协议来和服务端建立链接（TCP 三次握手）
  3. 建立连接后，客户机发送一个请求给服务器，请求方式的格式为：统一资源标识符（URL）、协议版本号，后边是 MIME 信息包括请求修饰符、客户机信息和可能的内容
  4. 服务器接到请求后，给予相应的响应信息，其格式为一个状态行，包括信息的协议版本号、一个成功或错误的代码，后边是 MIME 信息包括服务器信息、实体信息和可能的内容

https 加密原理

HTTPS 协议使用公钥加密技术来加密数据，并使用数字证书来验证服务器的身份。

HTTPS 加密过程

当客户端（浏览器）和服务器（网站）建立连接时，客户端会向服务器发送一个包含其支持的加密算法和协议版本的列表。服务器从中选择一种加密算法和协议版本，然后将自己的证书发送给客户端。

客户端收到服务器的证书后，会验证证书的有效性。如果证书有效，客户端会生成一个随机数，并使用服务器的公钥对其进行加密。然后，客户端将加密后的随机数发送给服务器。

服务器收到加密后的随机数后，使用自己的私钥对其进行解密。然后，服务器使用解密后的随机数和客户端的公钥生成一个会话密钥。会话密钥是客户端和服务器之间通信时使用的加密密钥（对称加密）。

客户端和服务器使用会话密钥对数据进行加密和解密。这样，即使数据在传输过程中被截获，也不能被解密。

tcp 怎么保证消息的可靠性

TCP（Transmission Control Protocol）通过多种机制来保证消息的可靠性：

  1. 序列号与确认应答：TCP 在传输数据时会对每个数据包进行编号，接收方会确认已收到的数据包。如果发送方没有收到确认应答，它会重新发送相应的数据包，直到接收到确认。
  2. 重传机制：如果发送方在合理的时间内没有收到确认应答，就会认为数据包丢失，并进行重传。这确保了即使在网络出现问题时，数据也能够可靠地传输。
  3. 流量控制：TCP 通过使用滑动窗口（Sliding Window）机制来控制发送方发送数据的速率，以适应接收方的处理能力。这样可以避免发送方发送过多的数据导致接收方无法处理，从而造成丢包或拥塞。
  4. 拥塞控制：TCP 通过拥塞窗口和慢启动算法来防止网络拥塞。它动态调整数据发送的速率，以确保在网络出现拥塞时能够减少数据包的丢失和重传。
  5. 校验和：TCP 在数据包的首部添加了校验和，用于检测数据在传输过程中是否发生了改变或损坏。如果接收方检测到数据包有错误，它将丢弃这个数据包，并通知发送方重新发送。
  6. 连接管理：TCP 建立了可靠的连接，包括三次握手和四次挥手的过程，确保了通信的可靠性。

总的来说，TCP 利用这些机制来确保数据的可靠传输，通过序列号、确认应答、重传、流量控制、拥塞控制、校验和和连接管理等手段，保证了数据在传输过程中的完整性、可靠性和有序性。

tcp 的阻塞控制

TCP 使用阻塞控制（Congestion Control）来避免网络拥塞，确保网络中的各个节点能够有效地处理数据并避免数据丢失。以下是 TCP 中阻塞控制的一些关键概念和机制：

  1. 慢启动（Slow Start）：

    • TCP 初始时采用慢启动算法，它限制了发送端最初发送数据包的数量。发送方开始时以较小的发送窗口（发送数据的数量）开始发送数据，并在收到确认应答后逐渐增加发送窗口的大小，实现指数级的增长。
    • 慢启动使得发送方在开始时以保守的方式发送数据，以便更好地适应网络环境。

  2. 拥塞避免（Congestion Avoidance）：

    • 一旦发送窗口达到一定阈值（通常是慢启动阈值），TCP 切换到拥塞避免阶段。在这个阶段，TCP 使用线性增长的方式来增加发送窗口的大小，而不是指数级增长。这有助于避免突然发送大量数据导致网络拥塞。

  3. 快速重传（Fast Retransmit）：

    • 当发送方连续收到相同的确认应答时，它会认为某个数据包丢失并进行快速重传，而不必等待超时时间。这可以更快地恢复丢失的数据包，从而减少网络拥塞的可能性。

  4. 拥塞窗口（Congestion Window）：

    • TCP 使用拥塞窗口来限制在任何时候可以发送到网络的数据量。拥塞窗口的大小受到网络拥塞程度的影响，根据网络拥塞的情况进行动态调整。

  5. 快速恢复（Fast Recovery）：

    • 在发生数据包丢失时，TCP 进入快速恢复状态。在快速恢复状态下，发送方会将拥塞窗口减小，并在一定条件下继续增加，而不是直接返回慢启动状态。这有助于更快地恢复到正常的发送状态。

总体而言，TCP 的阻塞控制通过这些算法和机制动态地调整数据发送的速率，以避免网络拥塞并确保数据的可靠传输。这些控制机制在 TCP 协议中起到重要作用，使其成为一种可靠且广泛使用的网络传输协议。

tcp 的请求过程

tcp 快速恢复，快速启动，慢启动

TCP 使用了几种算法和机制来优化数据传输的性能和可靠性，其中包括慢启动、快速启动和快速恢复等。

  1. 慢启动（Slow Start）：

    • 慢启动是 TCP 连接初始化时的一种算法，用于确定网络的可用带宽和拥塞情况。
    • 发送方在初始时，发送窗口大小被设置为一个较小的值。然后，每当收到一个确认应答时，发送窗口大小就会成倍增加。这种指数级增长可以更快地适应网络的带宽。
    • 这个阶段的目标是快速达到网络的合适发送速率，但也避免了可能的拥塞。

  2. 快速启动（Fast Start）：

    • 快速启动是一种优化算法，用于加速连接的初始阶段。它允许发送方在连接开始时一次性发送一定数量的数据，而不是按照慢启动的方式逐渐增加发送窗口大小。
    • 快速启动允许发送方根据网络的状况发送更多的数据，以便更快地达到适当的发送速率。

  3. 快速恢复（Fast Recovery）：

    • 快速恢复是 TCP 在出现数据包丢失时的一种优化机制。当发送方接收到连续的重复确认应答时，它认为有一个数据包丢失，并执行快速恢复算法。
    • 在快速恢复阶段，发送方将拥塞窗口减少一半，并转入拥塞避免状态。然后，它继续发送数据，而不是退回到慢启动状态，以便更快地恢复发送速率。

这些算法和机制使得 TCP 连接能够更有效地利用网络资源、避免网络拥塞并在丢失数据包时更快地恢复。通过慢启动、快速启动和快速恢复，TCP 连接可以在建立和维护期间更快地适应网络环境，提供更高效的数据传输。

tcp 中 time_wait 和 close_wait 具体干什么

TIME_WAIT

当客户端主动关闭连接时，它会向服务器发送一个 FIN 报文。服务器收到 FIN 报文后，会进入 CLOSE_WAIT 状态。此时，服务器会等待客户端发送一个 ACK 报文，以确认服务器的 FIN 报文已被收到。收到客户端的 ACK 报文后，服务器会进入 TIME_WAIT 状态。

TIME_WAIT 状态的目的是为了确保客户端和服务器之间的连接已经完全关闭。在 TIME_WAIT 状态期间，客户端和服务器都不能再发送或接收数据。TIME_WAIT 状态的持续时间通常为 2MSL（最大报文生存时间）。

CLOSE_WAIT

当服务器主动关闭连接时，它会向客户端发送一个 FIN 报文。客户端收到 FIN 报文后，会进入 CLOSE_WAIT 状态。此时，客户端会等待服务器发送一个 ACK 报文，以确认客户端的 FIN 报文已被收到。收到服务器的 ACK 报文后，客户端会进入 TIME_WAIT 状态。

CLOSE_WAIT 状态的目的是为了确保客户端和服务器之间的连接已经完全关闭。在 CLOSE_WAIT 状态期间，客户端和服务器都不能再发送或接收数据。CLOSE_WAIT 状态的持续时间通常为 2MSL。

进程是怎么通信的

进程间通信（IPC）是指不同进程之间交换信息和数据的一种机制。在 Linux 系统中，进程间通信可以通过以下几种方式实现：

管道

管道是一种单向的通信机制，它允许一个进程向另一个进程发送数据。管道通常用于父子进程之间的通信。

有名管道

有名管道是一种持久化的管道，它可以在不同的进程之间共享。有名管道通常用于不同进程之间的通信。

套接字

套接字是一种双向的通信机制，它允许两个进程在不同的机器上进行通信。套接字通常用于网络通信。

消息队列

消息队列是一种存储消息的机制，它允许进程在不同的时间发送和接收消息。消息队列通常用于进程之间异步通信。

共享内存

共享内存是一种进程之间共享内存的方式。共享内存通常用于进程之间快速交换数据。

信号

信号是一种通知进程的机制，它允许进程在发生特定事件时采取相应的行动。信号通常用于进程之间同步。

进程间通信的注意事项

在使用进程间通信时，需要注意以下几点：

  • 进程间通信可能会导致死锁。
  • 进程间通信可能会导致数据丢失。
  • 进程间通信可能会导致性能下降。

线程是怎么通信的

线程间通信（IPC）是指不同线程之间交换信息和数据的一种机制。在 Java 中，线程间通信可以通过以下几种方式实现：

共享变量

共享变量是一种最简单、最直接的线程间通信方式。共享变量是指多个线程都可以访问的变量。线程可以通过读写共享变量来进行通信。

锁

锁是一种同步机制，它允许线程在访问共享资源时进行同步。锁可以防止多个线程同时访问共享资源，从而避免数据不一致的情况。

条件变量

条件变量是一种同步机制，它允许线程在满足特定条件时被唤醒。条件变量通常与锁一起使用，以实现更复杂的线程间通信。

消息队列

消息队列是一种存储消息的机制，它允许线程在不同的时间发送和接收消息。消息队列通常用于线程之间异步通信。

管道

管道是一种单向的通信机制，它允许一个线程向另一个线程发送数据。管道通常用于父子线程之间的通信。

套接字

套接字是一种双向的通信机制，它允许两个线程在不同的机器上进行通信。套接字通常用于网络通信。

synchronized 和 volatile 的区别

synchronized 关键字是 Java 中的一个同步机制，用于保证对共享资源的访问是原子的。当一个线程进入一个 synchronized 代码块或方法时，它会获得该代码块或方法所属对象的锁。其他线程在该线程释放锁之前不能进入该代码块或方法。

volatile 关键字是 Java 中的一个内存可见性修饰符，用于保证对共享变量的访问是可见的。当一个线程修改了一个 volatile 变量时，其他线程能够立即看到该修改。

synchronized 和 volatile 都是 Java 中的并发控制机制，但它们的作用不同。synchronized 用于保证对共享资源的访问是原子的，而 volatile 用于保证对共享变量的访问是可见的。

hashtable、 hashmap 和 ConcurrentHashMap 的区别

  1. ConcurrentHashMap 对整个桶数组进行了分割分段(Segment)，然后在每一个分段上都用 lock 锁进行保护，每一把锁只锁容器其中一部分数据，多线程访问容器里不同数据段的数据，就不会存在锁竞争，提高并发访问率。（默认分配 16 个 Segment，比 Hashtable 效率提高 16 倍。）  到了 JDK1.8 的时候已经摒弃了 Segment 的概念，而是直接用 Node 数组+链表+红黑树的数据结构来实现，并发控制使用 synchronized 和 CAS 来操作。
  2. Hashtable(同一把锁) :使用 synchronized 来保证线程安全，效率非常低下。当一个线程访问同步方法时，其他线程也访问同步方法，可能会进入阻塞或轮询状态，如使用 put 添加元素，另一个线程不能使用 put 添加元素，也不能使用 get，竞争会越来越激烈效率越低。
  3. HashMap 没有锁机制，不是线程安全的。（JDK1.8 之后 ConcurrentHashMap 启用了一种全新的方式实现,利用 CAS 算法。）
  4. HashMap 的键值对允许有 null，但是 HashTable 和 ConcurrentHashMap 都不允许。（为什么？）

CAS 算法

对 CAS 的理解，CAS 是一种无锁算法，CAS 有 3 个操作数，内存值 V，旧的预期值 A，要修改的新值 B。当且仅当预期值 A 和内存值 V 相同时，将内存值 V 修改为 B，否则什么都不做。

CAS 缺点

  1. ABA 问题（加版本号 或 标志）
  2. 循环时间长开销大（破坏掉 for 死循环 或 等待一段时间）
  3. 只能保证一个共享变量的原子操作（用锁 或 把多个共享变量合并成一个共享变量来操作 或 封装成对象）

HashMap 为什么不直接使用 hashCode()处理后的哈希值直接作为 table 的下标？

hashCode()方法返回的是 int 整数类型，其范围为-(2 ^ 31)(2 ^ 31 - 1)，约有 40 亿个映射空间，而 HashMap 的容量范围是在 16（初始化默认值）2 ^ 30，HashMap 通常情况下是取不到最大值的，并且设备上也难以提供这么多的存储空间，从而导致通过 hashCode()计算出的哈希值可能不在数组大小范围内，进而无法匹配存储位置

ThreadLocal

ThreadLocal 适用于每个线程需要自己独立的实例且该实例需要在多个方法中被使用，也即变量在线程间隔离而在方法或类间共享的场景。

ThreadLocal 造成内存泄漏的原因？

ThreadLocalMap 中使用的 key 为 ThreadLocal 的弱引用,而 value 是强引用。所以，如果 ThreadLocal 没有被外部强引用的情况下，在垃圾回收的时候，key 会被清理掉，而 value 不会被清理掉。这样一来，ThreadLocalMap 中就会出现 key 为 null 的 Entry。假如我们不做任何措施的话，value 永远无法被 GC 回收，这个时候就可能会产生内存泄露。ThreadLocalMap 实现中已经考虑了这种情况，在调用 set()、get()、remove() 方法的时候，会清理掉 key 为 null 的记录。使用完 ThreadLocal 方法后 最好手动调用 remove()方法。

CyclicBarrier：通过它可以实现让一组线程等待至某个状态之后再全部同时执行。叫做回环是因为当所有等待线程都被释放以后，CyclicBarrier 可以被重用。我们暂且把这个状态就叫做 barrier，当调用 await()方法之后，线程就处于 barrier 了。

mysql buffer pool 工作原理

MySQL 的 Buffer Pool 是其主要的内存缓存机制，用于存储索引和数据页，以加速数据库的访问和查询。以下是 MySQL Buffer Pool 的工作原理：

  1. 缓存数据页：

    • Buffer Pool 是一个内存区域，用于缓存数据库中的数据页。数据页是数据库中索引或表的块，MySQL 以页的形式读取和写入数据。

  2. 缓存策略：

    • 当查询需要读取索引或表中的数据时，MySQL 首先检查 Buffer Pool。如果数据页已经在 Buffer Pool 中，MySQL 可以直接从内存中读取，而不必去磁盘读取，提高了访问速度。
    • 如果需要的数据页不在 Buffer Pool 中，MySQL 将其从磁盘读取到 Buffer Pool 中，并在内存中进行操作和修改。

  3. LRU 算法：

    • Buffer Pool 使用 LRU（Least Recently Used，最近最少使用）算法来管理数据页的加载和替换。
    • 当 Buffer Pool 满了并且需要加载新的数据页时，LRU 算法会淘汰最久没有被访问的数据页，腾出空间给新的数据页。

  4. 内存管理：

    • Buffer Pool 的大小可以通过 MySQL 的配置进行调整，以适应不同的内存环境和需求。
    • 它利用了系统的物理内存，因此需要足够的内存容量来存储缓存的数据页。

  5. 性能优化：

    • 通过合理配置 Buffer Pool 的大小，可以优化数据库的性能。如果 Buffer Pool 太小，系统可能频繁地访问磁盘，影响性能；而如果太大，可能会占用过多的内存资源。

总的来说，MySQL 的 Buffer Pool 通过在内存中缓存数据页，减少了频繁读写磁盘的操作，提高了数据库查询和访问的性能。优化 Buffer Pool 大小和合理使用内存是提高数据库性能的重要因素之一。
